# Reference Assemblies

KitsuneCommand is a 7DTD mod, which means at compile time it depends on six DLLs from the game itself:

- `Assembly-CSharp.dll` — the bulk of 7DTD's game logic
- `Assembly-CSharp-firstpass.dll` — early-loaded utility types
- `LogLibrary.dll` — 7DTD's logging facade
- `UnityEngine.dll` — Unity's engine API
- `UnityEngine.CoreModule.dll` — Unity's core types
- `0Harmony.dll` — the Harmony patching library 7DTD ships with

Those DLLs are **proprietary game assets**. We can't commit them to a public repo. But we still need *something* committed so that CI (and any contributor who doesn't own a 7DTD copy) can compile the mod.

The solution is **stripped reference assemblies**, generated by [JetBrains Refasmer](https://github.com/JetBrains/Refasmer).

## What's a stripped reference assembly?

A copy of the original DLL with:
- ✅ All public + protected type signatures preserved (so code compiles against them)
- ❌ All IL bodies replaced with `throw null` (so the file is tiny + can't be run)
- ❌ Internal/private members omitted

These files live at `src/KitsuneCommand/refs/`. They're roughly **30–50% the size of the originals** and contain **zero copyrighted IL** — they're just the publicly-documented API surface that any third-party developer can inspect via `ildasm` or Reflection.

At compile time, the C# compiler binds against these stripped DLLs. At runtime, the mod loads inside the real 7DTD process, which has the real DLLs in memory. The runtime never loads our stripped copies — they exist purely as a compile-time crutch.

## When to regenerate

Run `tools/strip-refs.{sh,ps1}` whenever:

- **7DTD ships a new release** that adds, removes, or changes the public API of types KC references
- **You add a new `<Reference>` to a csproj** — update the `$refs` array in the script first, then regen
- **You upgrade your local 7DTD install** and you're seeing diff noise from version-stamp differences

In practice this is rare — most 7DTD patches don't touch the API KC depends on. v2.7.x has been running against the same refasm output as v2.5.x.

## How to regenerate

### Prereqs

1. .NET SDK 6+ (you already have this for KC dev)
2. Refasmer CLI:
   ```bash
   dotnet tool install --global JetBrains.Refasmer.CliTool
   ```
3. Your local 7DTD dedicated-server install with `7DaysToDieServer_Data/Managed/` present
4. The six game DLLs copied from `Managed/` into `src/KitsuneCommand/7dtd-binaries/` (which is gitignored)

### Run

```bash
# Linux / macOS / Git Bash
bash tools/strip-refs.sh
```

```powershell
# Windows PowerShell
tools\strip-refs.ps1
```

The script:

1. Confirms `refasmer` is on PATH (errors out with install instructions if not)
2. Confirms all six source DLLs exist at the expected `7dtd-binaries/` path
3. Runs `refasmer --omit-non-api-members=true` over each
4. Prints a size summary (original → stripped, percentage ratio)

### After

```bash
git status
git diff --stat src/KitsuneCommand/refs/
```

If the diff is just byte-level changes in 1–2 DLLs, that's normal — refasmer's output isn't 100% deterministic across runs even on the same input. **Stage the diff and commit.**

If the diff is more dramatic (new DLLs appearing, big size jumps), 7DTD's API has changed substantially — be ready to fix any compile errors that surface from the new types.

## Verifying CI uses refs (not 7dtd-binaries)

The csproj `<HintPath>` for every game DLL points at `refs\<name>.dll`. To confirm:

```bash
grep HintPath src/KitsuneCommand/KitsuneCommand.csproj | grep -v 'lib\|Microsoft\.'
```

Every hit should reference `refs\`. Same for `KitsuneCommand.Tests`, `KitsunePvPBalance`, `KitsuneTraderUnlock` csprojs.

CI build (windows-latest runner) has no access to the real DLLs, so a successful CI build proves the refs/ DLLs are sufficient for compilation.

## What if I want the real DLLs locally?

You can keep them at `src/KitsuneCommand/7dtd-binaries/` for inspection (ILSpy, dotPeek, IDA, your IDE's "Go to Definition" navigation). That directory is in `.gitignore` so it won't accidentally land in a commit. The build doesn't use it — only the regen script does.

If you want your IDE to navigate into actual game source (where you can read the implementations), point your decompiler at `7dtd-binaries/Assembly-CSharp.dll` directly. The refs/ versions are for the compiler, not for reading.

## Why minisign-style trust signals don't replace this

Different problem. Code signing (kanban #138) lets a downloader verify the release zip matches what GitHub built. Reference assemblies let the build machine compile the mod against game types it doesn't own a copy of. Orthogonal.
